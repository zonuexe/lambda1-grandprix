# ドメインドキュメント

エンジニアリング系 skill が、コードベースを探索する際にこのリポジトリのドメインドキュメントをどう読むべきかを定めます。

## 探索前に読むもの

- ルートの **`CONTEXT.md`**、または
- ルートに **`CONTEXT-MAP.md`** があればそれ（コンテキストごとの `CONTEXT.md` を指す）。トピックに関連するものをそれぞれ読む。
- **`docs/adr/`** — これから作業する領域に関わる ADR を読む。マルチコンテキストリポジトリでは `src/<コンテキスト>/docs/adr/` のコンテキスト固有の判断も確認する。

これらのファイルが存在しない場合は **黙って続行** します。不在を指摘したり、先回りして作成を提案したりしないでください。`/domain-modeling` skill（`/grill-with-docs` や `/improve-codebase-architecture` 経由で到達）が、用語や判断が実際に確定したときに遅延生成します。

## ファイル構成

単一コンテキストリポジトリ（このリポジトリはこちら）:

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-....md
│   └── 0002-....md
└── ...
```

マルチコンテキストリポジトリ（ルートに `CONTEXT-MAP.md` がある場合）:

```
/
├── CONTEXT-MAP.md
├── docs/adr/                          ← システム全体の判断
└── src/
    ├── ordering/
    │   ├── CONTEXT.md
    │   └── docs/adr/                  ← コンテキスト固有の判断
    └── billing/
        ├── CONTEXT.md
        └── docs/adr/
```

## 用語集の語彙を使う

出力がドメイン概念に言及するとき（issue タイトル、リファクタ提案、仮説、テスト名など）は、`CONTEXT.md` で定義された用語を使います。用語集が明示的に避けている同義語へずらさないでください。

必要な概念がまだ用語集にない場合、それは合図です。プロジェクトが使っていない言葉を発明しているか（見直す）、本当のギャップがあるか（`/domain-modeling` 用に記録する）のどちらかです。

## ADR の矛盾を指摘する

出力が既存の ADR と矛盾する場合は、黙って上書きせず明示的に指摘します:

> _ADR-0007（イベントソース化された注文）と矛盾する — ただし次の理由で再検討の価値あり…_
